### **この質問がきたら必殺技**

| 質問 | 必殺技 |
|------|--------|
| 「このアプリで工夫した点は？」 | 「複数の設計判断があります。【1】単一方向データフロー（state を App に集約）【2】ドメイン型の活用（ObservationPointType）【3】計算ロジックのテスト化。特に 3 つ目は、現場計算との一致度を保証するために不可欠でした」 |
| 「バックエンド経験は？」 | 「このアプリはまだ localStorage ですが、拡張を想定しています。Node.js + PostgreSQL で REST API を実装し、複数ユーザー・複数現場に対応させる計画です。設計時から utils 層を分離してるので、API 連携に切り替えるのは容易です」 |
| 「なぜ React 選んだ？」 | 「SPA で UI の再計算が頻繁（フォーム入力 → 即座に結果表示）だからです。jQuery では state 管理が煩雑になるので、React の state/props でデータフローを明確化できるメリットが大きかった」 |
| 「他に作ったアプリある？」 | 「このプロジェクトに集中しました。1 つを深掘りして、『なぜこうしたのか』を説明できる方が、多数の小作品より評価が高いと考えています」 |

### パターン B：「なぜこの技術選んだ？」

```
【React】
Q：「Vue or Angular じゃダメ？」
A：「SPA で state 管理が必須。React は学習リソースが豊富で、
    業界標準。React を極めることが転職市場での価値だと判断」

【TypeScript】
Q：「JavaScript でいいでは？」
A：「計算エラーが信用失につながるので、コンパイル時に型チェック必須。
    意図しない値（"invalid" という文字列）を 防ぐため」

【Vite】
Q：「create-react-app は？」
A：「CRA は deprecated。Vite は高速ビルド、modern tooling。
    今新規プロジェクト なら Vite が標準」

【localStorage】
Q：「なぜバックエンド作らない？」
A：「MVP（最小実行可能製品）として。応用範囲：
   - 100 件程度なら十分
   - 拡張時に API に切り替え可能な設計
   - まずはローカルで動かしたい」
```
