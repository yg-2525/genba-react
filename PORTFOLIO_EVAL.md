# genba-react ポートフォリオ評価 & 面接対応ガイド

## 📊 ポートフォリオとしての総合評価

### **市場評価: 非常に優秀（★★★★★ 4.9/5.0）**

### 最新更新（2026-04-01 反映）

#### 機能面
- 現場名 + 日付で river.go.jp 公開データから水位を取得（Cloudflare Worker で CORS 対応）
- 開始時間・終了時間を入力した場合の「時間平均水位」取得に対応
- 水深から器深を自動算出（≥ 0.5m: 2割/8割、0.48-0.49: 6割で 0.30、< 0.48: 6割）
- 観測点タイプの表記を V/H/P に統一し、水深入力パターンから自動判定
- フローティングラベル + 入力済み非表示の洗練された UI
- 10分刻みカスタム TimeSelect コンポーネント

#### ドメイン精度
- **全計算値を切り上げ（Math.ceil）に統一** — 現場実務では切り上げが基本。点流速・平均流速・断面積・流量・平均水位に適用
- **秒数平均・距離は小数第1位まで** — 現場の実務機器の精度に合わせた桁数設定
- **器深の 0.05 刻み丸めも切り上げ** — 0.52 → 0.55, 0.48 → 0.50（6割で 0.30）

#### インフラ・品質
- **PWA 対応** — Service Worker + manifest.json でオフラインキャッシュ、ホーム画面追加可能
- **ErrorBoundary** — 予期しないクラッシュ時にリカバリ UI を表示
- **CSP (Content-Security-Policy)** — script-src / connect-src を制限し XSS リスクを軽減
- **localStorage ランタイム検証** — `isGembaData()` 型ガードで不正データを安全にフォールバック
- **API エラーサニタイズ** — 外部 API の生エラーメッセージをユーザーに露出しない
- **npm audit 0 脆弱性** — 依存関係に既知の脆弱性なし

### **Q2. アーキテクチャの特徴を説明してください**

**回答例（技術的に深い）:**
```
このアプリは、以下のレイヤー分離を意識しています：

【UI Layer (components/)】
- InputForm, EditModal, DataList, ChartView など
- React の state/props で状態を受け取り、ユーザーアクションをコールバック経由で App へ通知
- 計算ロジアを持たず、純粋に表示と入力に専念

【Business Logic Layer (utils/)】
- observation.ts: 観測データの計算エンジン
  * ObservationPointType 型で観測地点の3パターン（measured/depthOnly/pier）を表現
  * calcPointVelocity() で流速計算
  * calculateObservationSummary() で集計・集計
  * resolveRowVelocity() で欠損値補完
- date.ts, filterData.ts など純粋関数として再利用可能

【State Management (App.tsx)】
- 単一の main state: dataList（GembaData[]）
- localStorage と同期し、ブラウザ再起動後も永続化
- 検索・フィルタは useMemo で最適化

【データ型 (types.ts)】
- GembaData: 画面に表示・保存する観測記録
- ObservationInputRow/ObservationComputedRow: 入力行と計算済み行を分離
- CalculationSettings: 流速計の個体差（係数・オフセット）を扱うための型

この設計により、計算ロジックをテストしやすく、
UI 変更時に他への影響を最小化できます。
```

### **Q3. TypeScript をどの場面で活かしていますか？**

**回答例:**
```
3つの場面で強く活かしました：

1. ドメイン型の定義
   ObservationPointType = 'measured' | 'depthOnly' | 'pier'
   という型を定義することで、無効な観測地点タイプの属性を
   コンパイル時に防いでます。
   仮に 'invalid' を指定したら、即座に TS エラーが
   出るので、実装者のミスを早期に発見できます。

2. 計算関数の安全性
   calcPointVelocity(count: number, ...) と型を明記することで、
   呼び出し時に必ず number を渡す必要があります。
   文字列をうっかり渡すと type error になります。

3. State の型推論
   localStorage から取得したデータを JSON.parse すると
   any 型になりますが、as GembaData[] で型を明示することで
   以降の操作が全て type-safe になります。

4. ランタイム型ガード
   localStorage のデータは外部入力と同等なので、
   isGembaData() という型ガード関数を実装しました。
   JSON.parse 後に各プロパティの型を検証し、
   不正データが混入していた場合は安全にフォールバックします。
   コンパイル時の型安全 + ランタイム検証の二重防御です。

実装時に「あ、これって null かもしれない」という気づきや
「この計算結果は小数点第3位までの精度が必要」という要件を
型で表現できるので、バグを事前に防げました。
```

### **Q5. React を使う際に工夫した点は？**

**回答例:**
```
【パフォーマンス最適化】
- useMemo を使ってフィルタ結果をメモ化
  毎回フルリストをフィルタするのではなく、
  searchName や filterStartDate が変わった時だけ再計算

【コンポーネント分割】
- InputForm, DataList, ChartView を独立したコンポーネント化
  各コンポーネントが単一責任を持ち、テストしやすい

【レスポンシブ設計】
- useIsMobile() カスタムフックで画面幅を検知し、
  PC: 日付は<input type="date">（ネイティブピッカー）
  モバイル: 日付は<select>で年月日を分割入力
  という UX を実現
  
【Context API 活用】
- ToastContext で通知機能を全コンポーネントから利用可能に
  UI に notification システムを実装せずに済んでいます

【State 設計】
- App.tsx に唯一の state (dataList) を配置
  各コンポーネントは props 経由で値を受け取り、
  イベント時にコールバックで App に変更を通知
  これにより state の流れが一方向になり、
  バグの原因が特定しやすくなります

【セキュリティ・堅牢性】
- ErrorBoundary（クラスコンポーネント）で予期しないクラッシュをキャッチし、
  ユーザーに再読み込みボタンを表示
- CSP ヘッダーで XSS リスクを軽減
- localStorage データをランタイム型ガードで検証し、
  不正データ混入時も安全にフォールバック

【PWA 対応】
- Service Worker で stale-while-revalidate キャッシュ戦略を実装
- API 呼び出しは network-only でリアルタイム性を担保
- manifest.json でスマホのホーム画面追加に対応
```

   - Node.js + Express で REST API を作成

## 6️⃣ よくある質問への短回答集

| Q | A |
|---|---|
| **なぜ React なの？** | SPA の再レンダリングが必要。jQuery では state 管理が煩雑 |
| **Next.js は使わなかった？** | Static Site Hosting（GitHub Pages）なので SSR 不要。SPA で十分 |
| **GraphQL、Apollo は？** | バックエンドがないので導入時期ではないと判断 |
| **データベース何使う？** | PostgreSQL 想定（地理情報・日時型が充実。Sequelize で ORM） |
| **デプロイはどこに？** | GitHub Pages（開発版）。本番は AWS S3 + CloudFront や Vercel 検討 |
| **なぜ PWA にしたの？** | 現場は電波が不安定。オフラインでも使えることが必須要件 |
| **セキュリティは？** | CSP + localStorage 型ガード + エラーサニタイズ + ErrorBoundary の多層防御 |
| **CORS はどう対処した？** | Cloudflare Worker で CORS プロキシを実装。オリジン・パス・メソッド制限付き |
| **スケルト型を使わなかった？** | micro-interactions を優先。skeleton より即座の計算結果表示を重視 |

## 8️⃣ AI 活用について（面接対策）

### 前提：今の採用市場の現実

2026 年現在、採用担当は「これ AI で作ったんじゃない？」と当然考えます。
隠すのは最悪手。「使っていません」は嘘になる可能性が高く、信頼を失います。

**正解は「積極的に使ったが、自分の判断で作った」と言うこと。**

---

### Q. 「AI（Copilot / ChatGPT）使いましたか？」

**模範回答（最も加点されるパターン）:**
```
はい、GitHub Copilot を積極的に活用しました。

ただし、AI が出したコードをそのまま貼り付けたわけではありません。
具体的には、以下のように使い分けました：

【AI に任せたこと】
- ボイラープレート（フォームの基本構造など）の高速生成
- CSS のレイアウト微調整
- テストコードの雛形生成

【自分で判断・実装したこと】
- 観測計算式（切り上げルール、器深の 2割/8割 vs 6割 判定）
  → AI は「四捨五入」を提案したが、現場実務では切り上げだと知っていたので修正した
- 水深 0.48・0.49 の特殊ケース
  → 丸めると 0.50 になるが実務では 6割。このドメイン知識は AI にはない
- Cloudflare Worker のセキュリティ設計（オリジン制限、パス制限）
- 「CSP で inline script がブロックされる」問題の解決
- Service Worker のキャッシュ戦略の選択（stale-while-revalidate vs network-first）

つまり、AI は「書く速度を上げるツール」として活用し、
「何を作るか」「どう作るか」の判断は自分で行っています。

実際、AI が提案したコードが現場の実務ルールと違っていたケースが何度かあり、
それを見抜けずに修正できたのは、ドメイン知識があったからです。
```

---

### なぜこの回答が加点されるのか

採用担当が本当に知りたいのは「使ったかどうか」ではなく、
**「この人は自分で考えられるのか？」** です。

| 評価ポイント | このアプリでの証明 |
|---|---|
| AI の出力をそのまま使わない判断力 | 切り上げ vs 四捨五入の判断、0.48/0.49 の特殊処理 |
| ドメイン知識で AI を補完 | 現場計算の精度ルールは AI が知らない領域 |
| AI を生産性ツールとして活用 | ボイラープレート生成で開発速度を向上 |
| デバッグ能力 | CSP ブロック、SW キャッシュ問題など AI 以外のデバッグが必要な場面も解決 |

### その他の聴かれ方と対応

**Q. 「じゃあ AI なしで同じもの作れますか？」**
```
時間はかかりますが、作れます。
ただ、現実の開発現場で AI を使わない選択肢はもうないと思っています。
IDE や Excel の時代に「手書きでコード書けますか？」と同じで、

実際、AI が提案したコードには現場計算のドメイン知識が欠けていたので、

自分にドメイン知識があったからです。

AI は「一般的に正しいコード」は書けますが、

```

### AI 活用をポジティブに見せる 3 つのポイント

1. **「使った」と正直に言う** — 隠すと発覚時のダメージが大きい
2. **「AI が間違えたところを修正した」具体例を出す** — 切り上げ / 0.48 の例は最強
3. **「AI を使ってもドメイン知識がないと正しく作れない」と結論づける** — 自分の価値を証明
