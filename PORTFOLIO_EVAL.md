### **Q1. このアプリを作ろうと思った動機は？**

**回答例（良い例）:**
```
土木・測量の現場経験があり、現地での簡易計算が非常に手作業で非効率であることに気づきました。

そこで React で「リアルタイムに計算結果を表示できるアプリ」を実装することで、

ことを目標に開発しました。

技術面では、複雑な計算ロジックを TypeScript で型安全に実装し、

```

**避けるべき回答:**
- ❌ 「ポートフォリオが必要だから適当に作った」

### **Q2. アーキテクチャの特徴を説明してください**

**回答例（技術的に深い）:**
```
このアプリは、以下のレイヤー分離を意識しています：

【UI Layer (components/)】
- InputForm, EditModal, DataList, ChartView など
- React の state/props で状態を受け取り、ユーザーアクションをコールバック経由で App へ通知
- 計算ロジアを持たず、純粋に表示と入力に専念

【Business Logic Layer (utils/)】
- observation.ts: 観測データの計算エンジン
  * ObservationPointType 型で観測地点の3パターン（measured/depthOnly/pier）を表現
  * calcPointVelocity() で流速計算
  * calculateObservationSummary() で集計・集計
  * resolveRowVelocity() で欠損値補完
- date.ts, filterData.ts など純粋関数として再利用可能

【State Management (App.tsx)】
- 単一の main state: dataList（GembaData[]）
- localStorage と同期し、ブラウザ再起動後も永続化
- 検索・フィルタは useMemo で最適化

【データ型 (types.ts)】
- GembaData: 画面に表示・保存する観測記録
- ObservationInputRow/ObservationComputedRow: 入力行と計算済み行を分離
- CalculationSettings: 流速計の個体差（係数・オフセット）を扱うための型

この設計により、計算ロジックをテストしやすく、
UI 変更時に他への影響を最小化できます。
```

### **Q3. TypeScript をどの場面で活かしていますか？**

**回答例:**
```
3つの場面で強く活かしました：

1. ドメイン型の定義
   ObservationPointType = 'measured' | 'depthOnly' | 'pier'
   という型を定義することで、無効な観測地点タイプの属性を
   コンパイル時に防いでます。
   仮に 'invalid' を指定したら、即座に TS エラーが
   出るので、実装者のミスを早期に発見できます。

2. 計算関数の安全性
   calcPointVelocity(count: number, ...) と型を明記することで、
   呼び出し時に必ず number を渡す必要があります。
   文字列をうっかり渡すと type error になります。

3. State の型推論
   localStorage から取得したデータを JSON.parse すると
   any 型になりますが、as GembaData[] で型を明示することで
   以降の操作が全て type-safe になります。

実装時に「あ、これって null かもしれない」という気づきや
「この計算結果は小数点第3位までの精度が必要」という要件を
型で表現できるので、バグを事前に防げました。
```

### **Q4. テストはどのように書きました？**

**回答例:**
```
Vitest を用いて、主にビジネスロジック（utils 層）をテストしました。

【テスト戦略】
- 単体テスト（Unit Test）: observation.ts の計算関数
  例）calcPointVelocity(11, 23.6, settings) === 0.388
  Excel の手計算結果と一致するか検証
  
- 統合テスト（Integration Test）: calculateObservationSummary()
  複数の観測行から、最終的な流量値まで一気に計算し、
  中間値と最終値が正しいか確認
  
- エッジケーステスト: depthOnly/pier といった特殊な観測地点が
  正しく処理されるか

【テスト数】
現在 34 テスト。主に:
- 計算ロジック（4 tests）
- フィルタ・検索（8 tests）
- 統計パネル（5 tests）
- React コンポーネント（13 tests: 入力フォーム、データ一覧等）

【なぜテストを書いたのか】
現場計算と異なる結果が出るとユーザーの信頼を失うので、
Excel との一致性を自動検証できる環境が必須でした。
また、小数点の丸め方を変えると結果が変わるため、
リグレッションテストの重要性を実感しました。
```

### **Q5. React を使う際に工夫した点は？**

**回答例:**
```
【パフォーマンス最適化】
- useMemo を使ってフィルタ結果をメモ化
  毎回フルリストをフィルタするのではなく、
  searchName や filterStartDate が変わった時だけ再計算

【コンポーネント分割】
- InputForm, DataList, ChartView を独立したコンポーネント化
  各コンポーネントが単一責任を持ち、テストしやすい

【レスポンシブ設計】
- useIsMobile() カスタムフックで画面幅を検知し、
  PC: 日付は<input type="date">（ネイティブピッカー）
  モバイル: 日付は<select>で年月日を分割入力
  という UX を実現
  
【Context API 活用】
- ToastContext で通知機能を全コンポーネントから利用可能に
  UI に notification システムを実装せずに済んでいます

【State 設計】
- App.tsx に唯一の state (dataList) を配置
  各コンポーネントは props 経由で値を受け取り、
  イベント時にコールバックで App に変更を通知
  これにより state の流れが一方向になり、
  バグの原因が特定しやすくなります
```

### **Q6. 今後の拡張案を聞かせてください**

**回答例（加点になる）:**
```
さらに実用的にするための拡張は以下を想定しています：

1. バックエンド連携
   - Node.js + Express で REST API を作成
   - PostgreSQL に観測データを永続化
   - 複数の現場・複数ユーザーに対応

2. 認証機能
   - JWT を使ったユーザー認証
   - 現場担当者ごとの権限管理

3. PDF レポート生成
   - jsPDF を
使って、観測成果をワンクリックで PDF 出力
   - QR コード埋め込みで現場photo トレーサビリティ

4. オフライン対応
   - Service Worker で localStorage → サーバー同期
   - 現地で通信が断絶していても記録可能

5. 音声入力
   - ボイスメモ（秒数計測など）をリアルタイマーで記録
   - 現場での両手操作を軽減

技術選定理由:
- バックエンド: Node.js なら js フロント/バックで知識流用できる
- DB: PostgreSQL は地理情報（latlong）サポートが充実
- PDF: jsPDF より react-pdf の方が component-first で拡張性高い
```

## 6️⃣ よくある質問への短回答集

| Q | A |
|---|---|
| **なぜ React なの？** | SPA の再レンダリングが必要。jQuery では state 管理が煩雑 |
| **Next.js は使わなかった？** | Static Site Hosting（GitHub Pages）なので SSR 不要。SPA で十分 |
| **GraphQL、Apollo は？** | バックエンドがないので導入時期ではないと判断 |
| **データベース何使う？** | PostgreSQL 想定（地理情報・日時型が充実。Sequelize で ORM） |
| **デプロイはどこに？** | GitHub Pages（開発版）。本番は AWS S3 + CloudFront や Vercel 検討 |
| **スケルト型を使わなかった？** | micro-interactions を優先。skeleton より即座の計算結果表示を重視 |

- `.gitignore` で `node_modules` と `dist` を除外しており、不要成果物の混入リスクは低い
