## 🎯 よく聞かれる React 質問

### **Q. 「このコンポーネント設計にした理由は？」**

**想定背景:**
採用担当が見ているのは、App.tsx が主要な state を管理し、
各コンポーネントが props で値を受け取り、コールバックで通知する流れ。

**良い回答:**
```
このアプリの核は「観測データの CRUD」と「複雑な計算」です。

【データの流れ】
1. InputForm で新規データを入力
2. 計算エンジン（utils/observation.ts）で自動計算
3. App が dataList に追加
4. localStorage に同期
5. DataList が表示を更新

この流れを「単一方向フロー」にすることで何が得られるかというと：
- バグの原因が特定しやすい
  （データが複数方向に流れていると、どこで化けたか追跡困難）
- state の重複がない
  （親と子で同じ値を管理する二重管理を避ける）
- テストがしやすい
  （InputForm が onAdd コールバックを呼ぶと、
   その先は App 内のテストで確認できる）

昔のバージョンでは InputForm 内で localStorage 更新していたんですが、
それだと EditModal から編集した時に InputForm の処理が走らず、
データ不整合が起きました。

それ以来「状態管理は App に一元化する」というベストプラクティスを
採用しています。
```

**避ける回答（NG）:**
```
❌ 「Redux を使わず、useState だけで十分だと思いました」
   （Redux が不要な判断は OK だが、その理由を述べてない）

❌ 「コンポーネント分割は MVVMパターンに従っています」
   （Web では MVVM より「単一方向データフロー」が強い）
```

### **Q. 「なぜ ObservationCalculator という別コンポーネント作ったの？」**

**良い回答:**
```
観測地点の入力が複雑だからです。

元々は InputForm 内に日付入力＋観測地点入力ターブル がビタッと
入っていたんですが、観測地点の部分が「行追加、行削除、行をクリック時に
ビジュアルが切り替わる」という複雑な状態管理を持つようになりました。

それを InputForm に全て詰め込むと：
- 1 ファイルが 500 行を超える
- 観測計算ロジック（depthOnly/pier の処理）が混在する
- テストも「InputForm 全体」を立ち上げないと検証できない

なので「観測地点入力＆計算」を ObservationCalculator に切り出して：
- ObservationCalculator は「観測行の入力」に専念
- InputForm は「全体フォーム」に専念
- utils/observation.ts で「計算ロジック」を独立

これで関心事の分離（Separation of Concerns）が実現できました。
```

### **Q. 「useMemo の使い方、どう判断した？」**

**良い回答:**
```
filteredList を計算するのに useMemo を使いました。

理由は：
- searchName, filterStartDate, filterEndDate が変わった時だけ再計算したい
- 件数が 1000 件超えると、毎レンダリングフィルタするのは遅い
- filteredList を DataList, StatsPanel に受け渡すので、
  値が変わらなければ子コンポーネントの再レンダリングが走らない

実装面白いのは、依存配列に [dataList, searchName, filterStartDate, filterEndDate]
を指定したことで、これらのいずれかが変わった時だけ
フィルタ関数が実行されます。

ただし、使い分けとしては：
- 【useMemo 使う】計算コスト高い（フィルタ、ソート、複雑な変換）
- 【useMemo 不要】数値演算や短い配列操作
- 【useCallback 使う】関数をコールバック props で渡す時

このアプリでは、userEffect で副作用（localStorage 保存）も
正しく分離しているので、state の依存関係が
複雑になるのを防いでいます。
```

## 🎯 TypeScript 系の質問

### **Q. 「ObservationPointType = 'measured' | 'depthOnly' | 'pier' ってなぜ?」**

**良い回答:**
```
現場の観測方法が 3 パターンあるからです：

1. measured（計測点）
   - 水深＆音数＆秒数を計測→流速を算出
   
2. depthOnly（水深のみ）
   - 水深だけ計測（流速計なし、または障害物で計測不可）→流速は補完

3. pier（ピア：橋脚）
   - ピアの存在：断面積計算に入るが、流速なし

もし Union 型を使わなければ：
```typescript
// ❌ 何でもありのアプローチ
const row = {
  pointType: string,           // ← "invalid" でも通る
  count: string | null | 0,    // ← 値が不明確
  seconds: any,                // ← 危険
}
```

すると、計算処理が if (row.pointType === 'measured')
のような文字列チェックだらけになり、
「unknown value」のバグが後で出ます。

Union 型にすることで：
```typescript
type ObservationPointType = 'measured' | 'depthOnly' | 'pier'
const row: { pointType: ObservationPointType }
// ← 'invalid' って書いたらコンパイルエラー
```

コンパイル時に無効値を排除できるので、
実行時の予期しないバグが減ります。

これが「堅い型設計」の利点です。
```

### **Q. 「as GembaData[] って型強制するのは危険じゃ?」**

**良い回答:**
```
はい、実は危険です。ただし、このケースでは許容しました。

```typescript
const saved = localStorage.getItem('genbaData')
return saved ? (JSON.parse(saved) as GembaData[]) : []
```

JSON.parse(saved) は any 型を返すので、as で GembaData[] にキャストしてます。

本来これは「保証なしに型強制する」ので、実行時バグのリスクがあります。

より安全な実装は「型ガード関数」を作ることです：
```typescript
function isGembaData(obj: any): obj is GembaData {
  return (
    obj && 
    typeof obj.id === 'number' &&
    typeof obj.name === 'string' &&
    typeof obj.waterLevel === 'number'
    // ... etc
  )
}

return saved ? [JSON.parse(saved)].filter(isGembaData) : []
```

ただ、このアプリではそこまでの堅牢性は不要と判断しました。
理由は：
- localStorage は同一ドメイン内の自分たちのアプリが書いたデータのみ
- 外部からの入力（API）ではない
- 小規模アプリなのでリスク許容度が高い

実務なら「ユーザー入力 API」には必ず型ガードを書きます。
```

## 🎯 計算ロジック関連の質問

### **Q. 「なぜこの計算式を選んだ?」**

**想定質問内容:**
面接官が小数点の丸め方、補完方法を指摘してくる。

**良い回答:**
```
計算の根拠：

## 流速計算式
```
点流速 = ((音数 × パルス係数) / 秒数平均) × 係数 + オフセット
```
これは「一般的な電磁流速計のキャリブレーション式」です。
Excel の水文観測マニュアルでも同じ式を使っています。

パラメータ（係数・オフセット）は流速計の個体差によって異なるので、
手入力で対応できるようにしました。

## 小数点の丸め方
```
round(value, digits)  // digits 位で四捨五入
```
例：round(0.3876, 3) = 0.388

理由：
- 秒数は 0.01 秒精度（小数点第 2 位）
- 流速は 0.001 m/s 精度（小数点第 3 位）
- 断面積は 0.01 m² 精度（小数点第 2 位）

というのが現場実務での慣例だからです。
Excel と結果を合わせるために、丸め位を厳密にしています。

## 欠損値の補完（resolveRowVelocity）
水深のみの行の流速を「隣接する計測点から線形補完」しました。

理由：
- メドレー法（区分法）では、各区分に代表流速が必要
- 水深のみ行は流速計がないので「計測不可」
- ただし、前後の計測点から推測ができる
  （一般的に流速は滑らかに変化する）

線形補完を選んだのは、
- Excel でも同じアプローチ
- 非線形（2 次関数）は過学習になりうる
- シンプルが最強

という判断です。
```

## 🎯 テスト関連の質問

### **Q. 「34 個のテストで十分？全カバレッジ目指さないの?」**

**良い回答:**
```
意識的に「クリティカルパスのみ」テストしました。

【テストしてる部分】
✅ calcPointVelocity（流速計算）→ 最重要、バグると信頼失う
✅ calculateObservationSummary（集計） → 複数行の計算結果
✅ フィルタロジック → 間違った件数を表示すると致命的
✅ React コンポーネント の入出力 → 画面が崩れないか

【テストしてない部分】
❌ CSS のピクセルパーフェクト → Designer 確認で OK
❌ ブラウザの localStorage API → ブラウザがテスト済み
❌ Chart.js の描画 → ライブラリがテスト済み
❌ routing（SPA なので hash routing のみ） → シンプルで不要

100% カバレッジを目指すと、
- テスト保守コストがアプリ同等になる
- 副作用のない純粋関数の部分ばかりテストになる
- 実際のリスク（計算エラー、API エラー）がテストされない

なので、「リスクが高い部分」にテストを集中させるのが
プロの実装です。
```

### **Q. 「なぜ React Testing Library でコンポーネントテスト?」**

**良い回答:**
```
UI の実装詳細ではなく「ユーザーが見える動作」をテストしたいからです。

React Testing Library（RTL） では：
```typescript
render(<InputForm onAdd={...} />)
const input = screen.getByLabelText("現場名")
input.value = "test-site"
user.click(screen.getByText("追加"))
expect(onAdd).toHaveBeenCalled()
```

これは「ユーザーが現場名入力 → クリック」という
実際の使用方法に近いテストです。

セレクタテスト（Enzyme など）なら：
```typescript
wrapper.find('.input-form__input').simulate('change', ...)
```
コンポーネント構造が変わると、セレクタが壊れます。

RTL なら、ユーザーが「現場名 label」を見つけて入力する
という自然な流れを再現するので、
リファクタに強いテストになります。
```

### **Q. 「500 件の観測データを記録しても大丈夫?」**

**良い回答:**
```
ローカルストレージの制限から、

【現在の仕様】
- localStorage: 5-10 MB
- GembaData 1 件あたり ≈ 300 bytes
- → 理論上 15,000-30,000 件まで保存可能

【実装上の考慮】
現在は dataList をメモリに全て持つので、
1000 件超えると：
- 初期ロード時に全データを JS メモリに読み込む
- フィルタ時に 1000 件以上ループ
- Chart.js で 1000+ データ点を描画

この時点で「フロントエンドだけでは限界」です。

【本番環境への移行】
500-1000 件程度なら localStorage で十分。
それ以上なら：
1. バックエンド（Node.js + PostgreSQL）構築
2. ページング（1 ページ 100 件）
3. インデックス検索（日付範囲で DB 側で絞る）
4. グラフは最新 30 日分のみ表示

という設計にシフトします。

今の実装は「ローカル運用向け」として、
拡張性を失わない設計（utils 分離、型安全）にしているので、
バックエンド追加時も大規模リファクタ不要です。
```

### **Q. 「セキュリティ対策は?」**

**良い回答:**
```
ローカルアプリ（SPA on GitHub Pages）なので、
一般的な Web セキュリティ脅威は発生しません：

❌ SQL インジェクション → DB がない
❌ CSRF → サーバー側 state がない
❌ XSS → React が HTML エスケープを自動化

ただし、データバックアップの観点から：
- 誤削除対策 → 削除前に確認モーダル
- localStorage 環境による紛失リスク → CSV エクスポート機能

本番で API サーバー追加する時に重要になる対策：
- 認証（JWT）
- CORS 設定
- レート制限
- 入力値バリデーション

都度対応できるようにします。
```
